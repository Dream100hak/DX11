## DX11 프로젝트 개선 제안서

이 문서는 현재 코드베이스를 기준으로, 유지보수성과 확장성을 높이기 위해 우선적으로 개선할 항목들을 정리한 문서입니다.

---

## 1) 아키텍처/구조 개선

### 1-1. 엔진 루프 가시성 강화
- 현재 진입점은 `EditorTool/Main.cpp`의 `GAME->Run(...)`인데, 실제 루프/메시지 처리 구현 위치가 코드 탐색에서 즉시 드러나지 않습니다.
- **개선안**
  - 루프 핵심(`Run`, `WndProc`, `InitInstance`, `Update`) 구현을 한 파일에 모으거나, 위치를 README에 명시
  - 프레임 순서(입력 → 업데이트 → 렌더 → Present)를 다이어그램으로 고정
- **효과**
  - 신규 기여자 온보딩 시간 단축
  - 루프 관련 버그 디버깅 속도 향상

### 1-2. Editor/Engine 경계 명확화
- `EditorTool` 코드와 엔진 런타임 로직이 일부 흐름에서 강하게 결합되어 있습니다(모델 로드/프리뷰/리소스 캐싱 경로 공유).
- **개선안**
  - `Engine`은 런타임 책임만, `EditorTool`은 편집/툴 책임만 갖도록 API 경계 재정의
  - 에디터 전용 헬퍼(프리뷰 생성, 임시 오브젝트 생성)를 별도 모듈로 분리
- **효과**
  - 빌드 타깃 분리(게임 런타임 vs 에디터) 용이
  - 테스트 범위 축소 및 책임 분리

### 1-3. 싱글턴 의존 축소
- `GAME`, `SCENE`, `RESOURCES`, `TOOL` 등 글로벌 싱글턴 접근이 광범위합니다.
- **개선안**
  - 신규 코드부터 DI(의존성 주입) 또는 Context 객체 기반으로 점진 전환
  - 최소한 인터페이스 계층(`IRenderContext`, `IResourceService`) 도입
- **효과**
  - 단위 테스트 가능성 증가
  - 런타임 교체(멀티씬, 서버툴, 헤드리스) 기반 마련

---

## 2) 렌더링 파이프라인 개선

### 2-1. Render Pass/Technique 체계 문서화 및 enum화
- 현재 `SetPass(1)`, `SetTechnique(2)` 같은 매직 넘버 사용이 보입니다.
- **개선안**
  - pass/technique를 enum 또는 strong type으로 대체
  - 각 pass의 목적(Shadow, GBuffer, Forward, SSAO, UI 등)을 표로 문서화
- **효과**
  - 렌더 순서 회귀(regression) 감소
  - 셰이더 교체 시 영향도 분석이 쉬워짐

### 2-2. 렌더 큐/정렬 정책 명시
- 현재 씬 렌더 호출 구조는 확인되지만, 불투명/투명/UI/디버그 렌더 순서 정책이 문서로 고정돼 있지 않습니다.
- **개선안**
  - RenderQueue 분리: Opaque, AlphaTest, Transparent, Overlay
  - 큐별 정렬 기준(거리, 머티리얼, 셰이더) 명시
- **효과**
  - 드로우콜 최적화 및 시각적 아티팩트 감소

### 2-3. GPU 리소스 생명주기 추적
- 모델 메시 버퍼 생성(`ModelMesh::CreateBuffers`)은 명확하지만, 해제/재생성 규칙 문서가 부족합니다.
- **개선안**
  - 리소스 상태 전이(Loaded/Uploaded/Released) 표준화
  - 디바이스 로스트/리사이즈/핫리로드 대응 체크리스트 추가
- **효과**
  - 메모리 누수/중복 업로드 방지
  - 장시간 에디터 사용 안정성 향상

---

## 3) 에셋 파이프라인(Assimp/모델) 개선

### 3-1. 변환 파이프라인 표준화
- `AsConverter`에서 `.mesh/.mat/.mmat/.clip`을 생성하지만 규약 문서가 부족합니다.
- **개선안**
  - 파일 포맷 명세서 작성(헤더, 버전, 필드, endian, 단위)
  - 폴더 규칙(모델명/텍스처명/애니메이션명) 강제
- **효과**
  - 포맷 변경 시 호환성 관리 가능
  - 파이프라인 자동화(CI 변환) 기반 확보

### 3-2. 버전 관리/호환성 계층 추가
- 직렬화 포맷에 버전 개념이 명시적으로 드러나지 않습니다.
- **개선안**
  - 바이너리 파일 시작부에 magic + version 도입
  - 로더에 하위 버전 호환 처리 분기 추가
- **효과**
  - 포맷 변경 시 기존 리소스 재변환 부담 완화

### 3-3. 에러 처리 강화(Assert 중심에서 복구 중심으로)
- 파일/리소스 로드 경로에서 `assert` 의존이 강합니다.
- **개선안**
  - 사용자 노출 가능한 에러 리포트(파일명, 원인, 복구 제안) 도입
  - 에디터에서는 실패 시 대체 리소스(placeholder mesh/material) 사용
- **효과**
  - 크래시 감소
  - 에셋 문제 디버깅 생산성 향상

---

## 4) 코드 품질/안정성 개선

### 4-1. 주석 처리된 대규모 샘플 코드 정리
- `EditorTool/EditorTool.cpp`에 모델/애니메이션 생성 샘플이 장기간 주석 상태로 남아 있습니다.
- **개선안**
  - 샘플은 `Samples/`나 테스트 씬 생성 유틸로 이동
  - 운영 코드 경로에서는 dead/commented code 제거
- **효과**
  - 실제 동작 경로 파악이 쉬워짐
  - 리팩터링 시 충돌 포인트 감소

### 4-2. 네이밍/오탈자 정리
- 예: `Hiearchy`(Hierarchy 오탈자), `Cashe`(Cache 오탈자) 등 일관성 이슈가 보입니다.
- **개선안**
  - 네이밍 컨벤션 문서 + 일괄 리네이밍 계획 수립
  - 신규 PR에서 컨벤션 체크
- **효과**
  - 검색성 및 코드 가독성 개선

### 4-3. 로깅/진단 계층 강화
- 현재 런타임 경로에 구조화된 로그 정책이 부족합니다.
- **개선안**
  - 로그 레벨(Info/Warning/Error/Fatal), 카테고리(Render/Asset/Scene) 도입
  - 에디터에서 필터 가능한 로그 창 연동 강화
- **효과**
  - 재현 어려운 이슈 추적 속도 향상

---

## 5) 테스트/검증 체계 개선

### 5-1. 스모크 테스트 씬 도입
- 최소 검증 시나리오(정적 메시, 스킨 메시, 애니메이션, 그림자, SSAO, 썸네일)를 자동 점검하는 장치가 필요합니다.
- **개선안**
  - `SmokeScene` 구성 + 실행 체크리스트 문서화
  - 빌드 후 핵심 렌더 기능 확인 항목 고정
- **효과**
  - 렌더 회귀 조기 발견

### 5-2. 에셋 로드 회귀 테스트
- 모델/클립 로드 실패는 배포 직전 발견되기 쉽습니다.
- **개선안**
  - 지정 폴더의 `.mesh/.mat/.clip` 일괄 로드 검증 툴 작성
  - 실패 리소스 목록 자동 출력
- **효과**
  - 컨텐츠 규모 증가 시 안정성 확보

---

## 6) 문서화 개선

### 6-1. README 계층 분리
- 현재 README 1개에 많은 내용을 담으면 금방 비대해집니다.
- **개선안**
  - `README.md`: 빠른 개요/빌드/실행
  - `docs/rendering-pipeline.md`: 렌더 패스/상수버퍼/드로우 흐름
  - `docs/asset-pipeline.md`: Assimp 변환/포맷/규칙
  - `docs/editor-workflow.md`: 에디터 사용 흐름
- **효과**
  - 문서 탐색성 향상, 신규 합류자 학습 비용 감소

### 6-2. 코드-문서 싱크 규칙
- 문서가 코드와 쉽게 어긋날 수 있습니다.
- **개선안**
  - “렌더 패스 변경”, “포맷 변경” 시 문서 업데이트를 PR 체크리스트에 포함
- **효과**
  - 장기 유지보수 품질 향상

## 7) 우선순위 실행 로드맵(추천)

### 단기 (1~2주)
- pass/technique 매직 넘버 enum화
- 주석 샘플 코드 정리
- 네이밍/오탈자 정리 1차
- README에서 핵심 문서 링크 구조 확정

### 중기 (3~6주)
- 에셋 포맷 버전 필드 도입
- 로더 에러 처리 개선(placeholder + 리포트)
- RenderQueue 정렬 정책 문서화 및 적용

### 장기 (6주+)
- 싱글턴 의존 점진 축소(DI/Context 전환)
- 테스트 씬 자동 검증 체계 구축
- 에디터/런타임 모듈 경계 강화
